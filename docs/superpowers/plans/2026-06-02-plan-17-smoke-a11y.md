# Plan 17 — Manual Smoke Checklist + a11y Baseline Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`).

**Goal:** The final M1 gate. Write a manual smoke checklist that maps 1:1 to the 10 acceptance criteria in `docs/M1-CHARTER.md` § 5, plus do an a11y audit pass that fixes the highest-value issues (missing labels, weak focus rings, color-only meaning, button vs link semantics). After this plan, the M1 release is **gated only by the user actually running the checklist**.

**Architecture:** Two work streams.
1. **Smoke checklist** — a single `docs/smoke/M1-RELEASE-SMOKE.md` markdown file with 10 numbered sections, each containing concrete click-by-click steps and expected outcomes.
2. **a11y baseline** — grep the existing UI components for missing `aria-label`s on icon buttons, add a global `:focus-visible` outline rule, fix any obvious contrast or semantic issues found in a single audit pass.

**Tech Stack:** No new deps. Markdown + CSS + small TSX edits.

**Spec:** [`../specs/2026-06-01-designbridge-redesign-design.md`](../specs/2026-06-01-designbridge-redesign-design.md) § 12 (release gate) + M1-CHARTER § 4.4 (a11y baseline) + § 5 (10 acceptance criteria).

**Scope boundary (out of plan):** NO full WCAG audit. NO screen-reader testing on production tools (M2). NO keyboard-trap fixes for the OAuth popup. NO right-to-left support. NO color-blind simulation testing. M1 a11y is "ergonomic for keyboard users + correct semantics for screen readers"; certified compliance is later.

/* Visible focus ring on keyboard navigation only */
*:focus-visible {
  outline: 2px solid var(--accent);
  outline-offset: 2px;
  border-radius: var(--radius-sm);
}
*:focus:not(:focus-visible) { outline: none; }

/* Screen-reader-only utility for off-screen labels */
.sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;

    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}

### TopBar.tsx — already has aria-label on hamburger. Verify, ensure mobile select also has aria-label (added in Plan 16). Add semantic landmark:

Wrap with `role="banner"` if not implicit on `<header>`. The existing `<header className="workspace__top">` is fine; `<header>` is implicitly `role="banner"` when in body.

### RightInspector.tsx — aria-label on close button is already present. Add `role="complementary"`:

```tsx
<aside className="workspace__right" data-collapsed={rightCollapsed} role="complementary" aria-label="詳細資訊">
```

### TurnBubble.tsx — thinking toggle: change to `aria-expanded`:

```tsx
<button
  onClick={() => setShowThinking(!showThinking)}
  aria-expanded={showThinking}
  aria-controls={`thinking-${turn.id}`}
  style={{ background: 'transparent', border: 'none', color: 'var(--text-muted)', cursor: 'pointer', fontSize: 11, marginBottom: 4, padding: 0 }}
>
  {showThinking ? '▼ 隱藏推理' : '▶ 顯示推理'}
</button>
{showThinking && (
  <div id={`thinking-${turn.id}`} className="bubble__thinking">{turn.aiResponse.thinking}</div>
)}
```

### PhaseIndicator.tsx — pulsing dot needs `aria-hidden="true"` (decoration). The phase label container should be `role="status"` + `aria-live="polite"` so screen readers announce phase changes:

```tsx
<div className="phase-indicator" role="status" aria-live="polite">
  <span className="phase-indicator__dot" aria-hidden="true" />
  <span>{PHASE_LABEL[state.phase] ?? state.phase}</span>
  …
</div>
```

For the council accordion items, add `role="region"` + `aria-label`:

```tsx
{state.council.map((c) => (
  <div
    key={c.persona}
    className={`council__item${state.activeCouncilPersona === c.persona ? ' council__item--active' : ''}`}
    role="region"
    aria-label={`${PERSONA_LABEL[c.persona]} 的意見`}
  >
    …
  </div>
))}
```

```markdown
# DesignBridge M1 — a11y Baseline (2026-06-02)

This document records the findings of the M1 a11y audit and what was fixed under Plan 17.
It is the baseline we hold M1 to; future plans should not regress these checks.

## Coverage

- All `/setup`, `/login`, `/projects`, `/projects/:id`, `/settings` routes scanned.
- Manual desktop keyboard navigation: Tab through every actionable, verify focus visible.
- Manual mobile (Chrome dev tools iPhone SE) tap target audit.
- DOM landmark inspection (banner / main / complementary / form).

## What was fixed

| Component | Issue | Fix |
|---|---|---|
| Composer textarea | iOS auto-zoom on focus (font < 16px) | `font-size: 16px` at ≤768px (Plan 16) |
| Hamburger / close icon buttons | Missing aria-label | aria-label added (Plan 17 Task 2) |
| Thinking toggle | No `aria-expanded` / `aria-controls` link | added (Plan 17 Task 2) |
| Phase indicator dot | Decorative dot announced as text | `aria-hidden="true"` + container `role="status" aria-live="polite"` (Plan 17 Task 2) |
| Right inspector | Missing landmark role | `role="complementary"` (Plan 17 Task 2) |
| Council persona items | No region labelling | `role="region"` + `aria-label` (Plan 17 Task 2) |
| Global focus visibility | Browser-default thin ring on dark bg | `:focus-visible` rule with accent outline (Plan 17 Task 1) |
| Reduced motion | Pulsing dot animates always | `@media (prefers-reduced-motion: reduce)` blanket override (Plan 17 Task 1) |

## What was NOT fixed (known M1 gaps)

| Component | Issue | Reason / Plan |
|---|---|---|
| Vue iframe preview content | Iframe content is third-party HTML — Tailwind defaults may have low contrast | Out of M1 scope — AI-generated content quality is M2 |
| OAuth popup | Native browser window — keyboard trap possible | Out of our control |
| xyflow graph nodes | Manual drag is mouse-only | xyflow upstream issue; M2 can add keyboard nav |
| Skill upload textarea | No syntax highlighting / live validation | Defer to M2 |
| Long transcripts | Scroll-to-bottom on every token | Acceptable per Plan 9; M2 polite-scroll |

## How to keep this baseline

Before any PR that touches workspace shell / chat / settings UI:

1. Tab through new actionable elements; confirm focus ring visible.
2. Inspect new icon-only buttons for `aria-label`.
3. Inspect new toggles / collapsibles for `aria-expanded`.
4. Run Chrome DevTools Lighthouse a11y on `/projects/:id` and `/settings`; score ≥ 95.

## Contrast spot-check

Dark theme tokens against `--bg-root` (#060d1a):

| Pair | Ratio | WCAG |
|---|---|---|
| `--text-primary` (#f1f5f9) on `--bg-root` | 17.6:1 | AAA |
| `--text-secondary` (#cbd5e1) on `--bg-root` | 13.5:1 | AAA |
| `--text-muted` (#94a3b8) on `--bg-root` | 7.4:1 | AAA |
| `--text-accent` (#e9d5ff) on `--accent-glass` (rgba 22% bg) | 11.8:1 | AAA |
| `--accent` (#7c5cbf) buttons (white on accent) | 5.9:1 | AA |

All primary text combinations exceed WCAG AA 4.5:1 minimum.
```

## Risks / Notes

1. **Smoke checklist is not automated**: per user instruction "不要playwright測試", we trade automation for documentation. The risk is that smoke regressions slip through silently. Mitigation: any PR that touches workspace/chat/settings must rerun the relevant smoke section.
2. **Lighthouse 90 threshold**: chosen because some xyflow-internal nodes flag generic warnings we can't fix from outside. A score of 95+ on `/setup`, `/login`, `/projects`, `/settings` is achievable; `/projects/:id` may sit at 90-92 due to xyflow.
3. **a11y baseline is a snapshot**: it documents 2026-06-02 state. Future plans MUST update this doc when they touch the listed components, OR open a new audit doc.
4. **The 4-step "How to keep this baseline"**: relies on developer discipline. M2 can add an axe-core test in vitest to enforce automatically.
