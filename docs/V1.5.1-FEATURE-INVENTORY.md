## § 0. 一句話定位

v1.5.1 是一個跑了一年的**生產級 AI UI 原型生成平台**：使用者上傳規格 / 圖 / URL → AI 4 人團隊討論 → 並行產生多頁面 HTML 原型 → 即時預覽 / 標註 / 匯出 React / Vue / Next / Nuxt / Figma。三個模式（顧問 / 設計 / 架構），完整的設定 UI（OpenCode/MCP/Skills/Users/OAuth），多人即時協作，版本歷史，品質評分。

M1 v2.0-alpha.1 是「跨模式記憶 + 視覺化 thinking」概念重做，**只覆蓋 v1.5.1 約 20% 的功能**。

## § 1. 認證與使用者管理

### 1.1 首次 setup 流程
- **無使用者狀態**：顯示 `SetupPage`，建立第一個使用者（自動 admin role）
- **無管理員密碼**：顯示「設定管理員密碼」表單（最少 4 字元 + 確認）
- 完成後 token 存 `sessionStorage['admin_token']` + `localStorage['pb-auth-token']`

### 1.2 使用者選擇 modal（任何需身分驗證的操作）
- `requireAuth()` 觸發 → 顯示使用者列表
- 點現有 user → `POST /api/auth/login` → 即時登入
- 「+ 建立新使用者」inline form → `POST /api/users` → 直接登入新建者
- 「取消（以匿名繼續）」按鈕

### 1.3 管理員密碼登入
- 已有 admin 密碼但未認證時，SettingsPage 顯示密碼框
- `POST /api/auth/verify`
- 失敗無重試限制（顯示 authError）

### 1.4 變更管理員密碼
- SettingsPage 內可展開的 form：當前密碼 / 新密碼 / 確認
- `POST /api/auth/change`，回傳新 token 並更新 sessionStorage
- 成功 toast「密碼已成功變更」（3 秒）

### 1.5 使用者列表（admin 限定）
- 表格 columns：名稱 / 角色（管理員/使用者徽章）/ 狀態（啟用/停用徽章）/ 建立時間 / 操作
- 操作：
  - **轉移管理員**（confirm dialog） → `POST /api/users/transfer-admin`
  - **停用/啟用** → `PATCH /api/users/:id/disable` 或 `/enable`
  - **刪除**（confirm dialog） → `DELETE /api/users/:id`

## § 4. AI 生成 Pipeline（核心）

### 4.1 Provider 路由（provider.ts）
- **3 層 fallback**：OpenCode（多 server）→ Gemini（key pool）→ OpenAI（OAuth 或 API key）
- **OAuth 自動刷新**：每 5 分鐘 poll，<10 分鐘到期時換新
- **Mid-stream retry**：`streamWithRetry()` 偵測 503/429/parse-error/network → 清緩衝 → 重來
- **Vision 自動選 model**：偵測 imageData → 升 visionModel()
- **JSON 模式**：`withJsonInstruction()` + `extractJsonBody()`（去 markdown fence）

### 4.2 4 人團隊討論（plannerAgent.ts）
| 角色 | 職責 | Token |
|---|---|---|
| Echo (PM) | 需求分析 / 找盲點 / 5-7 個頁面候選 | 10K |
| Lisa (UX) | 規劃頁面 + flow | 10K |
| David (QA) | 規則衝突 / 漏掉的場景 | 10K |
| Lisa 回應 | 防守 David | 6K |
| Bob (Tech) | 最終決策 | 10K |
| Echo 確認 | 導航完整性 / PAGES 清單 | 6K |

- 每個角色看到的 Skill 不同（role-keyword 配對）
- 5 分鐘 TTL 快取（pageNames 集合 → Skill 過濾結果）
- Layer 3 lessons（上次失敗紀錄）寫入每個 prompt
- SSE 即時串流給 client

### 4.3 規劃 fallback（buildLocalPlan）
- AI 失敗（429 overload）時離線產生
- 9 種網站範本（shopping/travel/education/medical/saas/news/restaurant/portfolio/event/realestate/library）
- 從 designConvention 文字挖 hex color → 衍生 light/dark
- 預設 320+ 行共享 CSS（flex/grid/card/button/form/nav）

### 4.4 並行頁面生成
- 每頁獨立 sub-agent 呼叫
- SSE event：`{phase: planning/tokens/assembling/generating, page, status}`
- 失敗頁面注入可見的「此頁面生成失敗」div

### 4.5 HTML 組裝（htmlAssembler.ts）
- 抽出嵌入 CSS → 移到 `<head>`
- 去掉 sub-agent 誤加的 nav/footer
- 替換外部圖片為 CSS placeholder
- 修 div 平衡
- 空頁 fallback
- 包裝 `<div class="page" data-page="X">`
- showPage(name) script

### 4.6 QA 驗證（htmlQaValidator.ts）
8 項檢查：空頁、孤島頁、div 失衡、白字白底、card 無 onclick、硬編碼 hex、導航無 active、表單縱版

### 4.7 品質評分（qualityScorer.ts）
5 維 0-100 分：html / accessibility / responsive / consistency / design / overall  
觸發時機：fire-and-forget，key pool 充足（≥10 keys）時才評

### 4.8 衝突偵測（skillConflictChecker.ts）
- 需求 vs Skill 規則比對
- 3 級：critical / warning / info
- SSE 推送 `conflict-report` event
- 失敗 graceful 回 `conflicts: []`

### 4.9 Intent 分類（intentClassifier.ts）
5 類：question / component / micro-adjust / full-page / in-shell  
5 分鐘 TTL 快取  
支援 vision（帶圖更精準）

### 4.10 Variant 選擇
- `POST /api/projects/:id/generate-variants` — 產 A/B/C 變體
- chat panel modal 顯示 thumbnails
- `POST /api/projects/:id/select-variant` 套用選定

### 4.11 單頁重生
- `POST /api/projects/:id/regenerate-page` — 保留其他頁

### 4.12 元素級重生 (Quick Regen)
- 點 preview 元素 → popup
- textarea 輸入指令 + 提示詞歷史 chips
- `POST /api/projects/:id/prototype/regenerate-component` (SSE)
- 收到 `{done:true, html}` → postMessage iframe `swap-component`

## § 6. 上傳與檔案分析

### 6.1 ChatPanel 上傳區
- Drag-drop area（dashed border，hover 高亮）+「選擇檔案」按鈕
- 支援格式：PDF / DOCX / PPTX / XLSX / CSV / TXT / MD / images
- 大小：≤ 20MB / 同送 ≤ 5 個
- 檔案 chip：name + 分類 intent 下拉 + 狀態徽章 + 進度條 / spinner + 刪除按鈕

### 6.2 5 階段非同步分析 pipeline（upload.ts）
1. **文字抽取**（同步）：
   - PDF → `pdf-parse`
   - DOCX → `mammoth`
   - PPTX → `yauzl` + XML 正規搜 `<a:t>`
   - XLSX → `xlsx` sheet_to_csv
   - Image → '' (留給 OCR)
   - TXT/CSV/MD → 直讀
2. **OCR**（fire-and-forget）：`tesseract.js` chi_tra+eng，60s timeout
3. **視覺分析**（async）：vision model → `visual_analysis`（pages/components/globalStyles JSON）
4. **藝術風格**（async）：PPTX/DOCX 圖片 → 1-2 句美感描述 → `art_style_preferences`
5. **文件分析 agent**（async）：classifyDocument → extractSpec/Design/Mixed → 4 個並行 agent（explore/uxReview/designProposal/businessContext）→ reviewSpecDocuments → `analysis_result` 完整 JSON

### 6.3 分析狀態 API
- `GET /api/projects/:id/upload/:fileId/analysis-status`（poll 每 2s）
- `GET /api/projects/:id/upload/spec-status`
- 重新分析：`POST /api/projects/:id/upload/:fileId/reanalyze` / `reanalyze-doc`

### 6.4 檔案標籤
- `PATCH /api/projects/:id/upload/:fileId/label` — 標 intent (design-spec/data-spec/brand-guide/reference) + component_label

### 6.5 分析彙總
- `GET /api/projects/:id/analysis-summary` — 整合所有檔案的 pages/globalRules/documentType

### 6.6 上傳檔案表（`uploaded_files`）關鍵欄位
- `extracted_text`, `visual_analysis`, `analysis_result`, `analysis_status`
- `component_label`, `intent`, `page_count`, `page_name`

## § 21. 結論

v1.5.1 是個**功能完整、生產驗證過**的平台。M1 v2.0-alpha.1 是「概念重做」，**只覆蓋核心對話流程**，砍掉所有：
- 管理員操作（OpenCode/MCP/Skills 批次/API key 批量/User 管理）
- 上傳檔案的深度分析（OCR/vision/文件 agent/藝術風格）
- 設計工具（URL crawler/全域設計/預設庫/平台 shell）
- Prototype 後操作（標註/API binding/約束/元件庫/品質評分/版本 diff）
- 匯出能力（React/Vue/Next/Nuxt zip + Figma）
- 即時協作的 cursor/lock

要把 M1 補齊到 v1.5.1 水準，估計需要 **15-20 個額外 plan**，工時相當於再做一次 M1 規模的工程。

**建議**：以 v1.5.1 為主版本繼續用，M1 的 turn-based 記憶池和 council 概念當作 v1.6 或 v2.0 的部分功能加進去，而不是完全取代。
